# 独立批判审查协议

本协议用于阻止三类伪深度：写得太少、写得很多但重复、资料复述却没有解释性综合。中度以上必须执行；用户明确要求深度、完整读完或真正学懂时最低按中度处理。

## 导航

- 角色隔离、冻结版本与分离输入
- 九道批判门、攻击测试与机器记录
- 判定规则与防橡皮图章

## 一、角色隔离

审查者不参与检索、综合和初稿写作。必须建立两个彼此隔离、也与作者隔离的审查上下文：

1. **证据/逻辑审查者**：看到研究说明书、深度契约、覆盖图、来源与主张台账、解释性综合记录、自动检查和冻结成稿；
2. **盲态读者审查者**：只看到冻结成稿、目标读者画像和一个预先设计的新情境任务；不得看到来源台账、综合记录、作者说明或证据审查结论。

两者不能使用同一个 reviewer/context 标识。这样避免拥有背景资料的专家自动补全正文缺失的逻辑，再误判初学者也能读懂。

审查者：

- 不能直接替作者重写整篇；
- 不能用总体印象或总分放行；
- 必须定位具体章节、主张或段落；
- 必须尝试推翻核心判断，而不是帮作者找理由；
- 每个问题要说明为何影响真实性、理解或决策；
- 只能输出 `PASS`、`REPAIR` 或 `BLOCKED`。

环境没有两个独立审查上下文时，中度以上状态为 `BLOCKED`。只有用户明确重新协商并接受降低档位或独立性时才能降级；作者自审可以作为修稿工具，但不能签发合同 `PASS`。

独立子 Agent 仍可能与作者使用相似模型，因此它降低的是同一写作上下文和角色造成的偏误，不能冒充真实第三方同行评审。关键事实仍须回到来源核验。

## 二、冻结版本与两阶段反锚定

审查前冻结候选版本，例如 `candidate-v1`。审查期间作者不得静默修改；修复后生成新版本并记录修改位置、关闭的问题以及是否改变结论、证据强度或适用范围。深度契约的 `candidate_files` 必须列出全部用户可见 Markdown、图片、图表和附件；Markdown 引用到的本地文件不在清单中时验证失败。`evidence_files` 列出证据/逻辑审查实际看到的来源、主张、覆盖、综合与自动检查文件。

两份审查记录必须写入候选 SHA-256、深度契约 SHA-256、契约版本、带时区的 ISO 审查时间和调度层返回的 reviewer/context 标识；证据/逻辑审查还必须写入 evidence bundle SHA-256，盲态读者记录不得包含证据包哈希。候选哈希对路径和文本做跨平台规范化（`/` 路径、Unicode NFC、UTF-8 BOM 与 CRLF/LF 等价），二进制资产按实际字节计；契约哈希按解析后的 JSON 键排序和 Unicode NFC 规范化。任何规范化后仍存在的正文、图片、证据台账或契约变化都会使旧 `PASS` 失效，必须复审。

批判审查分两阶段：

1. **预注册攻击面**：审查者先只看研究说明书、覆盖图、深度契约和证据包，不看候选稿结论；列出必须证明的决定性主张、最可能的因果跳跃、替代解释、口径/层级/版本风险、结论失效条件、高风险来源和陌生迁移案例。
2. **攻击冻结成稿**：再提供候选稿，按预注册清单逐项测试，并增加阅读成稿后发现的新问题。

如果工具环境不能分阶段发送材料，审查者必须先在输出中写出攻击清单，再开始评价成稿。不得因为成稿写得流畅、自信或引用很多而删除预注册攻击点。

## 三、分离的审查输入

证据/逻辑审查者必须获得：

1. 用户确认的研究说明书；
2. 动态覆盖图与深度契约；
3. 最终候选稿；
4. 来源台账和主张—证据台账；
5. 解释性综合记录；
6. 自动检查输出；
7. 已知证据缺口和资料冻结日期。

不提供作者的自评分和希望的审查结论。

盲态读者审查者只能获得：

1. 最终冻结稿；
2. 目标读者已有知识、陌生概念和实际用途；
3. 一个正文没有直接答案、但可以用核心模型解决的新案例。

如果盲态读者要求补充来源包、作者解释或正确答案，先记录它为何无法仅凭正文完成，再判定 `REPAIR`；不能把额外信息补给它后继续冒充盲测。

## 四、九道批判门

### 1. 任务追踪门

研究说明书中的每个核心问题在正文中有明确回答；没有用外围知识制造“看起来全面”。

### 2. 证据门

抽查全部决定性主张和高风险主张：引用真实可定位、实际蕴含、表述强度匹配、独立性成立。发现一项虚构来源或决定性证据错配即 `REPAIR`；若无法补证且影响核心结论则 `BLOCKED`。

### 3. 反复述门

核心章节不是按报告/论文/作者顺序改写。隐藏来源名称后，正文仍有清楚的关系模型、证据桥梁和条件化判断。

### 4. 解释门

每个核心模型能回答：为什么、凭什么、还用了什么前提、哪个替代解释最强、为什么当前判断更合理、什么条件会翻转。若证据只允许描述，正文诚实说明机制未知。

### 5. 批判与反证门

审查者主动构造最强反方、伪因果、同源证据、层级混淆和异常案例。正文必须处理会改变结论的反证，不能以“各有利弊”回避裁决。

### 6. 教学与迁移门（由盲态读者独立执行）

以目标读者身份复述核心模型，再应用到一个正文没有直接给答案的新案例。无法迁移说明读者得到的是结论记忆，不是理解。

### 7. 连贯写作门

段落和章节形成论证脊柱；不能随机交换顺序；列表、表格、图和 Callout 没有替代正文推理；结尾比开头增加了一个更完整模型。

### 8. 篇幅充分与反填充门

最终稿达到研究说明书中确认的动态篇幅下界；每个核心解释单元达到自己的最低解释容量。重复段落、同义改写、模板化小结、无关历史和来源堆叠不计入有效篇幅。低于下界直接 `REPAIR`；达到下界仍需通过其余八门。

### 9. 不确定性诚实门

事实、观点、综合、建议和未知可区分；局限不是装饰性免责声明，而是与受影响结论和翻转条件相连。

## 五、强制攻击测试

审查者至少执行与课题相关的六项：

1. **为什么/凭什么测试**：向下追问决定性结论；
2. **来源替换测试**：隐藏来源名检查是否仍有模型；
3. **最强反方测试**：用最有力而非稻草人的替代解释攻击；
4. **异常案例测试**：寻找模型最难解释的反例；
5. **同源证据测试**：检查多引用是否回到同一底稿；
6. **新案例迁移测试**：改变情境变量，检查模型是否能生成条件化判断；
7. **打乱顺序测试**：检查结构是否有真实依赖；
8. **删视觉测试**：删表格/图/Callout 后正文是否仍完整；
9. **反填充测试**：寻找重复和只增加字数、不增加理解的段落。

## 六、严重度与输出格式

问题严重度统一使用：

- `P0`：真实性、核心结论、任务完整性或审查独立性失败；
- `P1`：会让目标读者误解、无法迁移或关键边界缺失；
- `P2`：不阻塞结论但值得修复的表达或维护问题。

```markdown
# 独立批判审查

## 总状态
PASS / REPAIR / BLOCKED

## 硬门结果
| 门 | 状态 | 严重度 | 具体证据位置 | 问题为何重要 | 必须怎样修复 |
|---|---|---|---|---|---|

## 核心问题逐项追踪
| 核心问题 | 正文位置 | 证据 | 解释桥 | 反证/边界 | 迁移 | 状态 |
|---|---|---|---|---|---|---|

## 最强反方与结论翻转条件

## 篇幅与填充审计
- 动态下界：
- 实际有效篇幅：
- 重复/填充位置：
- 判断：

## 阻塞项（如有）

## 复审清单
```

每个 `REPAIR` 必须给出可验证的关闭条件，不能只说“再深入一点”“加强逻辑”。

除可读 Markdown 审查意见外，必须用 `assets/critical-review-record.template.json` 写出机器可核对记录。证据/逻辑审查需要完整记录八道对应硬门；盲态读者审查需要记录读者理解、机制复现、新案例迁移、连贯、术语负担和反填充六道门。每个 PASS 必须指向真实存在的 Markdown 标题并写出实质 finding；反填充记录要保存有效 NPU 和重复比例；所有 P0/P1 必须关闭并记录关闭证据。

validator 能验证记录结构、时间、位置、内容下限、哈希和两个 context 标识不同，但不能从一段 JSON 密码学证明某个 Agent 的真实身份。总控必须从实际子 Agent 调度结果抄录 context 标识，不能由作者随意创造；这一步是运行时硬门，不是静态脚本能够替代的事实。

## 七、判定规则

- **PASS**：九道门全部通过；自动检查无硬错误；所有 P0/P1 问题关闭；动态篇幅下界满足且未发现实质填充。
- **REPAIR**：存在可通过补搜、重建模型、重写解释或调整结构修复的问题。作者修复后，审查者依据原问题清单逐项复审，并检查修复是否引入新矛盾。
- **BLOCKED**：关键证据客观不可得、合同要求的独立审查能力不可用、或核心前提无法在当前范围内验证。不得把阻塞写成完整结论。

中度以上都必须完成证据/逻辑与盲态读者两个视角；可以由两个独立 Agent 并行完成，再由总控合并问题。修复后至少一次对新冻结版本的复审，不能由作者自称已关闭。

## 八、防止橡皮图章

- 不给综合总分；任何决定性硬门失败就不能被平均分抵消；
- 每个 PASS 必须指向正文或台账证据；
- 审查者至少提出一个最强反方和一个新案例；
- “写得很全面、结构很清晰”不是有效审查结论；
- 不规定“至少找出几条问题”，避免为了显得严格而捏造缺陷；首轮零问题时仍必须记录最强反方、来源替换和反填充测试的实际过程与结果；
- 审查记录默认保留为内部产物；用户要求过程透明时整理后交付。
