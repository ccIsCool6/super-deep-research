# 研究成果的教学写作标准

本文件规定 Deep Research 如何把已经完成的研究写成可理解、可学习、可复查、可迁移的成品。正式写作前必须先完成 `explanatory-synthesis.md` 的证据约束综合；不能一边按来源摘要，一边期待格式规则自动产生深度。

研究正确不等于读者学会。来源可靠、结论成立，只能说明材料有资格进入成品；读者能否形成正确的内部模型，取决于概念出现的顺序、关系是否显式、证据怎样嵌入解释、例子是否击中误解，以及文档是否允许分层阅读和以后重新进入。

## 导航

- 写作任务、读者模型与认知推进路径
- 完整解释单元、段落/列表/表格/视觉分工
- 证据进入解释、单/多文档教学架构
- 超级重度契约、两遍编辑与交付检查

## 一、先确认写作任务：帮助判断，还是帮助掌握

研究成品至少有两种不同使命。

### 决策型成品

读者需要在有限时间内判断、比较或行动。正文应先交代：结论是什么、依据是什么、适用条件是什么、什么情况会使结论改变。详细机制与证据放在随后章节或专题文档中。

### 掌握型成品

读者希望以后能独立解释、辨别和应用这个领域。正文不能只交付“正确答案”，而要逐步建立：

1. 研究对象到底是什么，不是什么；
2. 关键组成部分如何连接；
3. 机制为什么会产生观察到的结果；
4. 哪些证据支撑、限制或挑战这套解释；
5. 换一个情境后，读者如何自行推导；
6. 哪些地方仍然未知或存在真正分歧。

### 混合型成品

混合不等于把两套写法搅在一起。通常采用“双层结构”：主文档先给决策入口和全局地图，专题文档负责完整教学；或主文档先建立最小理解，再给建议，附录保留审计细节。必须在研究简报中指定主路径。

> [!important] 三种质量不是一回事
> **可读**表示句子容易读；**可理解**表示读者能解释概念与关系；**可掌握**表示读者能在新问题中调用这些关系。不要用短句、列表和漂亮排版冒充教学完成。

## 二、把“人类友好”当作设计问题，不当作格式配额

下列原则是写作判断依据，不是伪精确的排版定律：

- 读者会用已有知识解释新内容，因此必须声明预备知识，并从读者真实起点开始；
- 同时需要维持的陌生元素越多，理解越困难，因此应分阶段引入概念并及时说明关系；
- 长文的困难不仅是读完，还包括查找、比较、暂停后恢复和数周后重新进入；
- 关系若只存在作者脑中，读者就要自行猜测，所以因果、对比、层级、时序、条件和例外应被明确表达；
- 例子能降低抽象门槛，但单个例子也可能让读者误把偶然特征当本质，因此重要概念需要反例或边界例子；
- 视觉锚点能帮助定位，却也会竞争注意力；只有承担语义功能的表格、图、强调和 Callout 才值得保留。

不得写“人类一次只能记住四项，所以每个列表最多四项”之类未经语境限定的规则。段落长度、列表项数和标题密度只能作为复查触发器，不能代替语义判断。

## 三、先设计读者将形成什么模型

开始写作前，为每个文档填写内部的“教学契约”。它不一定呈现在成品中，但必须真实影响正文。

```markdown
- 本文回答的核心问题：
- 读者进入前已经知道什么：
- 读者最可能带着什么误解：
- 读完后应该能够解释什么：
- 本文必须建立的概念与关系：
- 一个最能暴露误解的例子或反例：
- 依赖哪些前置文档：
- 哪些内容明确留给其他文档：
```

若无法说清“读完后读者新增了哪种可迁移能力”，这个文档的边界通常还没有设计好。

## 四、选择符合课题的认知推进路径

“建立—打破—重建”只是可选路径，不是所有长文的固定骨架。根据读者的起点与知识形状选择。

### 基础 → 机制 → 应用

适合技术、科学、制度或方法体系。先定义对象与边界，再解释内部机制，最后进入案例、操作与失败条件。

### 现象 →  competing explanations → 证据裁决

适合争议、历史因果和社会科学。先把共同观察到的现象说清，再分别呈现竞争解释，最后依据证据区分它们能解释什么、不能解释什么。

### 案例 → 诊断 → 原理 → 新案例迁移

适合读者缺少抽象基础但有实际经验的主题。先用具体案例暴露问题，再抽取原理，随后换一个案例检查原理是否真正被理解。

### 对比对象 → 共同尺度 → 条件化选择

适合产品、方案、理论或政策比较。必须先建立共同评价维度，避免每一方使用不同标准自证优越。

### 历史变化 → 转折机制 → 当前格局

适合概念演化、产业、制度与思想史。时间线只展示“发生了什么”，正文要解释为什么那些转折改变了后续可能性。

### 常识模型 → 异常现象 → 修正后的模型

适合用户已有一个简化但不完全错误的理解。先承认旧模型在哪些条件下有用，再用它无法解释的现象推动修正；不要为了戏剧性制造虚假的“认知打破”。

同一知识库可以让不同文档采用不同路径，但主文档必须说明它们如何共同构成全局模型。

## 五、一个解释单元怎样写完整

重要概念、机制或判断通常需要以下要素。不是每段都要机械填满，但不可长期缺少其中决定理解的部分。

1. **对象**：这段到底在解释什么；
2. **定义与边界**：这个词在本文中怎样使用，哪些相邻概念不包含在内；
3. **关系或机制**：组成部分怎样相互作用，为什么会出现结果；
4. **证据**：观察、数据、文本或实验怎样支持这一步，而不只是贴一个引用；
5. **例子**：把抽象关系放进一个具体情境；
6. **反例或边界**：在哪些情况下不成立、变弱或需要换模型；
7. **含义**：这会改变读者后续的哪个判断；
8. **迁移**：面对相似但不相同的新问题，应该看哪些变量。

### 示例：从结论句改成教学段落

只有结论的写法：

> 多代理评审能提高 AI 产出质量。

教学写法应继续回答：多代理与“重复问三次”有什么区别；它通过独立错误、角色分工还是证据核验起作用；在共享同一错误来源、互相从众或缺少验证标准时为什么可能不增反降；什么任务值得付出额外成本；读者如何判断本次评审是否真的增加了独立信息。引用应放在它实际支持的那一步附近，而不是堆在段尾给整段背书。

> [!example] 例子承担什么职责
> 好例子不是把概念换成人名，而是让读者看到变量怎样变化。若要解释“独立评审”，应让两个评审拥有不同证据或不同检查任务；如果它们只看到同一草稿、收到相同提示并互相模仿，就不能借数量假装独立验证。

### 深度合格的六条抽象判据（交付前逐条自检）

“深度够不够”不能靠字数或锚点数量证明。以下六条是跨领域通用标尺；任何一条系统性不达标，都说明文档仍停留在结论或材料堆砌，需要回到解释性综合，而不是补篇幅。

1. **机制性**：解释了“为什么会这样”，而不止“是什么”。反检法——对每个核心结论追问一次“所以呢 / 凭什么”，如果答不出下一层原因，这条就还是断言而非解释。
2. **可检验性**：结论带着证据、来源或推理链，读者能自行判断真伪，而不是被要求信任。区分“来源事实 / 来源观点 / 研究综合 / 建议”四类语句，别把综合措辞伪装成事实。
3. **边界感**：说明了“在什么条件下不成立、会变弱或需要换模型”。没有边界的绝对断言默认可疑；给出适用范围本身就是深度的一部分。
4. **结构从属性**：章节与小标题是从推理里自然生长出来的，不是先搭一个通用模板再往格子里填内容。检验——若把各节顺序打乱而文章读起来没什么不对，说明结构没有承载真实依赖，需要重新组织而不是加过渡句。
5. **综合增量**：文章不按来源顺序串讲，而是建立了单个来源没有直接给出的关系模型；这个增量必须受证据约束，并公开推断前提和可翻转条件。
6. **迁移能力**：读者能够把模型用于一个正文没有直接给答案的新情境，指出需要观察的变量，而不是只复述作者结论。

> [!tip] 判据怎么用
> 这六条是语义判断，不是脚本能替你计算的。脚本只能提示重复、段落墙和格式风险，不能证明文章已经有深度。宁可结论少而每条都扛得住追问，不可结论多而层层悬空。

## 六、段落、列表、表格和图各自承担什么工作

### 段落：承担推理

一个段落通常围绕一个中心动作展开：提出判断、解释原因、处理反例或推出含义。句子之间必须能读出连接关系。若连续句子只是并列事实，应考虑它们是否需要综合；若不能综合，列表或表格可能更诚实。

不要把每句话都拆成 bullet。列表擅长枚举，不擅长证明。连续多屏列表会隐藏“哪些是原因、哪些是证据、哪些更重要、彼此如何冲突”。

### 列表：承担并列或步骤

只在成员确实平行时使用。每个列表前应有一句说明“这组项目共同回答什么”，复杂项目后应有综合句说明优先级或关系。若项目内部各自需要多段解释，应拆成小节，而不是制造巨型列表。

### 表格：承担重复字段比较

当至少三项对象沿同一组字段比较时，表格通常比重复段落更清楚。不要把长论证或完全不同结构的内容硬塞进单元格。表格之后要指出真正重要的差异，不能让读者自行从数据墙中猜结论。

### Mermaid 或其他图：承担关系结构

图适合依赖、流程、反馈回路、层级、分支和时间变化。若关系用一两句话已经清楚，图只会增加切换成本。图后必须用正文解释读者应该看见的关系和限制；图不能替代证据。

### 粗体：承担扫描锚点

粗体用于关键词、判断转折和局部结论，不用于整句铺满。一个段落里处处加粗等于没有重点。

### 锚点取舍：语义触发，不按字数配额

深度是主体，锚点是辅助——但“辅助到位”会让深度更容易被读者吸收，所以既不是越少越好，也不是越多越好。落到操作上：

- **按知识点触发，不按字数触发**：一个值得凝练的认知（一个关键关系、一次对比、一处机制、一段容易踩错的边界）配一个锚点。不设“每 N 字必须一个锚点”这种机械配额，那只会逼出为达标而硬塞的空锚点。
- **把两端当作复查信号**：长文没有块级锚点时，确认纯散文是否确实最清楚；锚点逼近自然段数量时，确认它们是否开始替代论证。两者都是警报，不是必须加图或删图的机械死线。
- **优先类型匹配而非数量堆叠**：关系结构用图、同维度多对象比较用表、旁路提醒用 Callout。重度文档若通篇只有一种锚点（例如全是 Callout、零表零图），先问“有没有一个关系其实更适合用表或图承载”，而不是再加几个同类锚点。

> [!warning] 防止矫枉过正
> 发现锚点过多而做减法时，不要一路删到接近零——那是从“喧宾夺主”滑向“结论平铺”的另一个极端。正确的动作是让每个锚点各司其职：留下真正承担语义的，删掉重复背书或纯装饰的。深度为主、锚点为辅，两者不是此消彼长，而是各在其位。

## 七、旁路信息与可选 Obsidian Callout

默认 standard Markdown 使用普通引用块并用加粗前缀说明语义，例如 `> **边界：** ...`。只有用户明确选择 Obsidian 配置时才使用下列 Callout。

Callout 是旁路信息，不是装饰盒。核心论证必须在普通正文中仍然成立；读者跳过 Callout 不应失去主线。

```markdown
> [!summary] 本节模型
> 用两三句话压缩刚刚建立的关系。

> [!example] 具体案例
> 展示抽象机制如何作用于真实情境。

> [!warning] 适用边界
> 说明常见误用、风险或会让结论失效的条件。

> [!question] 尚未解决
> 标出证据仍然不足或存在真实分歧的问题。

> [!tip] 如何使用
> 给出从当前原理推导出的实际检查方法。

> [!note]- 审计细节
> 折叠保存方法、定义差异或扩展证据，避免阻断主线。
```

使用规则：

- 同一种语义保持同一种类型；
- 不连续堆叠多个 Callout；
- 不把每节摘要都强制放进 Callout；
- 不把决定性证据、关键限定或唯一结论藏在折叠块里；
- 关键论证和限定留在正文；切换回 standard 配置时改为普通引用块和标准相对链接。

## 八、证据必须进入解释，而不是附着在结论表面

研究写作至少区分四类语句：

- **来源事实**：来源直接报告、测量或记载的内容；
- **来源观点**：作者、机构或受访者作出的解释；
- **研究综合**：研究者把多个来源连接后得到的判断；
- **建议**：基于事实、价值取舍和用户情境作出的行动主张。

重要推理应能回答三个问题：

1. 主张是什么；
2. 证据实际支持主张的哪一部分；
3. 从证据到结论还使用了什么推断或前提。

研究者综合必须进一步说明最强替代解释、当前裁决和翻转条件。不要输出私有逐步思维草稿；要输出读者可以复核的公开理由。

引用放在最小可归属的主张组附近。一个链接不能给整段多个不同事实“统一背书”。如果原始研究只证明相关，不得把综合措辞升级为因果；如果来源只覆盖特定版本、国家、人群或时间，应在正文而非脚注里保留限定。

## 九、多文档知识库怎样让人学，而不只是把文件拆开

### 00 号文档是地图，不是缩略正文

它至少承担：

- 用普通语言定义研究对象与边界；
- 给出全局模型或核心问题地图；
- 说明文档之间的依赖与推荐阅读路径；
- 为“快速判断”“从零学习”“按问题查阅”等不同读者提供入口；
- 链接每一个交付模块，并说明读者为什么要打开它；
- 告知资料冻结日期、主要不确定性和知识库覆盖不到的地方。

### 子文档是完整学习单元

子文档不能只是主文档某个 bullet 的扩写。它应有清晰核心问题、必要前置、完整解释、证据、例子/边界和回到全局模型的出口。开头说明本文在系统中的位置，结尾说明它改变了什么认识以及下一步该读哪里。

### 交叉链接表达真实关系

只有“上一篇 / 下一篇”不足以构成知识网络。涉及依赖、对照、例外、共同机制或证据冲突时，应在相关位置直接链接并说明关系。链接文字要描述目的，例如“若要理解为什么这一建议只适用于低风险任务，见……”，而不是裸写“相关阅读”。

## 十、超级重度的“接近领域掌握”完成契约

超级重度不承诺穷尽世界上的所有信息，也不能保证读者仅靠阅读就获得实践经验。它应承诺：在已确认边界、资料冻结日期和可获得证据内，提供一条足以建立高水平认知、继续核验并进入实践的完整路径。

除非课题本身不适用，成品应覆盖：

| 覆盖层 | 读者应获得什么 |
|---|---|
| 对象与边界 | 知道领域研究什么、不研究什么，关键术语有哪些竞争定义 |
| 基础与演化 | 理解当前格局从何而来，哪些历史转折仍在影响今天 |
| 组成与机制 | 能解释核心部件、因果链、反馈和约束怎样共同工作 |
| 证据与方法 | 知道结论依据什么方法得出，证据强弱怎样判断 |
| 主要流派与争议 | 能准确重述主要竞争观点及各自最强证据，而非稻草人化 |
| 实践与案例 | 看见理论如何进入真实情境，包括成功、失败和混合案例 |
| 工具与工作方法 | 知道专业人士怎样调查、判断、验证和更新结论 |
| 风险与边界 | 知道常见误用、失败模式、伦理/安全限制和结论翻转条件 |
| 当前前沿与未知 | 区分已经稳定的知识、正在变化的共识和真正未解的问题 |
| 继续学习路径 | 能从引用、术语索引和主题书目进入原始资料与更深分支 |

还应提供适量的“理解验证”：不是为了像学校考试，而是让读者发现自己是否只记住了词语。可使用新案例分析、边界判断、自我解释提示、对比题或一项小型实践任务。纯决策报告可不加；以掌握为目标的超级重度成品不得完全缺少迁移设计。

> [!warning] 不要虚假宣称“完全掌握”
> 文档可以让读者获得接近专家的概念地图、证据入口和判断方法，但经验、反馈、真实项目和同行校正仍不可由阅读完全替代。最终成果应明确它能替代哪些入门信息成本，不能替代哪些实践过程。

## 十一、两遍编辑，而不是边写边装饰

### 第一遍：语义与教学编辑

检查论点是否成立、概念是否按依赖顺序出现、关系是否被解释、证据是否蕴含主张、例子是否真的消除误解、边界与不确定性是否存在。必要时重写和重排内容。

### 第二遍：阅读与视觉编辑

在核心内容稳定后，执行 `research-post-scan.md`。寻找段落墙、伪列表、比较机会、可视化机会、缺少导航、锚点失真和引用拥堵。允许在不改变主张与证据含义的前提下拆段、转表、增加 Callout 或重排局部表达；一旦发现逻辑需要改变，返回第一遍编辑。

最后运行：

```bash
python3 scripts/scan_readability.py <交付目录>
python3 scripts/validate_deliverable.py <交付目录> --tier <medium|heavy|super-heavy> --profile standard --h1-policy error
python3 scripts/validate_research.py --sources <来源台账.csv> --claims <主张证据台账.csv> --tier <medium|heavy|super-heavy>
python3 scripts/validate_review.py --record <证据逻辑审查.json> --deliverable <交付目录> --contract <深度契约.json> --review-type evidence-logic
python3 scripts/validate_review.py --record <盲态读者审查.json> --deliverable <交付目录> --contract <深度契约.json> --review-type blind-reader
python3 scripts/validate_depth.py --contract <深度契约.json> --deliverable <交付目录> --tier <medium|heavy|super-heavy>
```

Windows 使用 `py -3` 并使用反斜杠脚本路径。自动扫描只负责发现高概率问题和确定性错误，不能证明文档已经可学。脚本通过后仍需人工完成证据蕴含、解释性综合、迁移与连贯性审查。
