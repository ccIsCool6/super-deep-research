# 研究执行工作流

本文件只负责用户确认《研究说明书》后的执行阶段。原则是：内部流程严密、遇到缺口自动返工；用户侧只在范围、证据契约或交付目标发生实质变化时被重新打断。

## 导航

- 启动门与十步主流程
- 长任务与多代理协作
- 停止条件

## 启动门

必须已经确认：真实问题、用途与读者、范围和排除项、时间/版本边界、研究档位、关键证据要求、解释性综合完成标准、动态篇幅下界、批判审查方式、交付容器和保存位置。

执行中若新证据推翻核心前提或迫使范围改变，暂停并重新对齐。普通检索选择和局部缺口由研究者自主处理。

## 主流程

### 1. 建立动态覆盖图

把目标转化为：必须回答的问题、决定性主张、必要背景、主要分歧、适用边界、读者可能的误解和明确排除项。每个节点标记重要性、风险、时效、证据类型与读者理解依赖。

覆盖图控制检索和写作，但不是固定章节目录。

若任务要求读完、完整解释或忠实重建指定长文档，先把原文目录、章节、方法附录、结论限定和重要图表拆成 `assets/document-coverage.template.csv`。记录每份来源的真实总页数；无页码材料建立稳定线性等价单元。每个条目最多跨 50 页/单元，并记录精确范围、覆盖状态、映射的核心问题/解释单元、允许省略的理由；全部页/单元都必须有状态。覆盖台账在综合前冻结；不能只精读有利章节后用流畅文章冒充“读完”。

这类任务分两层：第一层忠实说明原文实际论证了什么；第二层用外部证据判断它是否成立、遗漏了什么。只有用户要求第二层或结论风险需要核验时才扩展外部研究，且正文要把原文重建与外部裁决分开。

### 2. 规划证据路线

按 `evidence-protocol.md` 判断每类主张需要什么来源。加载 `depth-model.md` 当前档位的来源目标，将其作为研究预算和过早停止警报；真正的硬门是决定性主张覆盖和高风险主张独立核验。

### 3. 第一轮研究

先建立课题全景和来源生态，再深入决定结论的节点。同步记录：来源角色、实际支持的主张、方法与范围、来源家族、反证、版本和利益冲突。

搜索结果页、AI 摘要和未实际阅读的摘要只用于发现线索，不计为合格来源。

### 4. 缺口与反证搜索

形成初步判断后，专门寻找：

- 尚无可靠支撑的决定性主张；
- 与当前模型冲突的证据、失败案例和异常现象；
- 来源之间的定义、样本、方法、日期、版本或利益立场差异；
- 引用链最终回到同一底稿的伪独立证据；
- 会使结论翻转的条件；
- 当前结构遗漏的重要板块。

重度和超级重度至少执行一次独立的反证检索轮。每轮结束更新覆盖图与台账。

### 5. 解释性综合

在正式成稿前完整执行 `explanatory-synthesis.md`。不要边读边把来源摘要写进最终报告。

对每个核心问题形成：

1. 来源共同支持的观察；
2. 一致、互补和冲突材料之间的关系；
3. 能解释主要现象与异常的最小模型；
4. 证据到判断的公开理由桥与关键前提；
5. 最强替代解释及当前裁决；
6. 适用边界、翻转条件和未知项；
7. 对读者判断的含义；
8. 一个新情境中的迁移检验。

中度以上使用 `assets/explanatory-synthesis.template.md` 保留内部记录。核心字段未完成，或仍以“来源 A 说、来源 B 说”为主要结构时，不得进入正式写作。

### 6. 生成教学或论证架构

依据用户用途选择主路径：

- 决策论证：方向性结论/判断入口 → 证据与机制 → 选择条件 → 风险与翻转条件；
- 认知引导：读者起点 → 必要概念 → 机制/竞争解释 → 证据裁决 → 新模型与迁移；
- 混合：主报告给地图与判断入口，专题部分完成教学。

不强制“建立—打破—重建”，也不故意隐藏答案。结构由知识依赖和中心问题推进关系生成。

在动笔前冻结 `assets/depth-contract.template.json`：用户确认的每个核心问题映射到至少一个中央解释单元和唯一正文位置，脚本根据实际语义负荷计算单元下界和跨单元综合下界。至少存在一个中央单元，且中央单元不能省略机制、证据、替代解释、边界、案例、含义或迁移。写作中发现新核心问题时上调契约；降低总下界或任一中央单元下界超过 10% 都必须重新对齐。

技术学习任务还要在中央单元中规划 worked example、故障/反例轨迹、可复现检查与带答案的迁移练习；不能用长篇概念说明代替实际推演。

### 7. 正式成稿

以解释模型而不是来源顺序为写作骨架。事实、来源观点、研究综合、建议和未知项语义可区分。引用靠近它真正支持的最小主张组，正文解释证据为何支持当前判断。

单文档和多文档遵循同一语义标准；容器按 `artifact-output.md` 选择。

### 8. 全局总编

由一个总编统一处理全部文档：

- 修复跨章节矛盾、重复与术语漂移；
- 统一数字、日期、版本和比较口径；
- 检查概念是否在依赖前解释；
- 检查章节顺序是否承载真实推理依赖；
- 检查摘要没有强于正文证据；
- 检查结尾回应用户用途而非套固定模板；
- 执行来源替换、为什么、竞争解释、异常、新案例迁移与连贯性测试。

### 9. 独立批判审查

冻结候选稿，执行 `critical-review.md`。证据/逻辑审查 Agent 先根据研究说明书和证据包预注册攻击面，再获得成稿进行证据、逻辑、反复述、边界、篇幅和反填充攻击。另开独立盲态读者 Agent，只给成稿、读者画像和新案例，检查正文自身是否足以让目标读者复述机制并迁移。

两名审查者分别生成 `assets/critical-review-record.template.json` 记录。候选哈希覆盖 `candidate_files` 中全部用户可见文件和 Markdown 引用的本地资产；证据/逻辑记录另绑定 `evidence_files` 的 evidence bundle；两者同时绑定规范化契约 SHA-256 和契约版本。`REPAIR` 返回最近阶段并生成新候选版本；任何正文、图片、证据台账或契约变化都会让旧 PASS 失效，原审查者或新的独立审查者必须对新版本复验。`BLOCKED` 如实报告。作者不能自行签发审查 `PASS`。

### 10. 质量门与修复

按顺序执行：

1. 证据与主张审计；
2. 解释性综合语义审计；
3. 反证、边界、时效和适用范围审计；
4. 动态 NPU 下界、重复折损与两份哈希绑定的独立审查；
5. 盲态读者迁移、全局连贯与教学审计；
6. 阅读性扫描；
7. 文件、链接、编码、JSON/CSV 和占位符验证。

任何失败返回最近的正确阶段：证据缺口回到检索，解释桥缺失回到综合，结构断裂回到架构/总编，格式问题回到阅读编辑。修复后重新执行所有受影响的门。

## 长任务与多代理

重度和超级重度采用 artifact-first：先建立状态、覆盖图、来源台账、主张台账和最小成品骨架，再持续保存稳定成果。

多个研究分支能够独立产生可核验产物时可以并行；总控负责边界、共同术语和去重。各代理的结果只是研究材料，必须由一个总编完成解释性综合，禁止直接拼接成交付物。

## 停止条件

只有同时满足以下条件才可停止：

- 决定性主张覆盖；
- 高风险与争议主张达到独立核验要求；
- 反证、冲突、边界、未知和翻转条件得到处理；
- 达到证据饱和；
- 来源目标达到，或短缺原因与影响已经透明记录；
- 核心章节通过解释性综合与迁移测试；
- 各解释单元和总有效 NPU 达到冻结的动态下界，重复/近重复比例未超标；
- 独立批判审查通过并关闭全部非建议性问题；
- 全局总编和确定性检查通过。

来源更多、文章更长或文件更多均不能替代以上条件。
