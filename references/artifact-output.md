# 成品形态与跨平台输出

研究深度由证据、解释性综合和读者理解增量证明，不由文件数量证明。交付容器服务于阅读用途和知识形状。

## 导航

- 选择容器与默认 Markdown 配置
- 保存位置、内部产物与最终产物
- 单/多文档架构、跨平台约定与交付

## 一、选择容器

### 聊天回答

适合边界清楚、线性结构明显且不需要长期维护的成果。即使内部进行了重度研究，也只展示用户需要的成品，不倾倒日志和台账。

### 单文档

适合一条连续论证、正式报告、出版/分享稿或用户希望从头读到尾的学习材料。重度研究也可以单文档交付，但内部仍保留证据与综合审计。

### 多文档

适合多个可独立维护的认知模块、非线性查阅、多人协作或长期更新。按认知边界拆分，不按字数、来源数量或代理分工拆分。

默认建议：

- 轻度：聊天或单文档；
- 中度：主报告，可按需要附来源/证据附录；
- 重度：默认多文档；中心论证明显时可单文档；
- 超级重度：默认多文档领域知识工程。

用户用途高于默认建议。交付方式在研究说明书中确认，执行中不静默改变。

## 二、默认 Markdown 配置

### 标准配置（默认）

- UTF-8 `.md`；
- 标准 Markdown 标题、列表、表格、代码块、图片和普通引用块；
- 文档间使用标准相对链接，如 `[核心机制](./02-核心机制.md)`；
- 文件名避免 Windows 保留名（如 `CON`、`AUX`、`COM1`）、`<>:"\|?*`、控制字符、末尾空格/句点和仅大小写不同的冲突；
- 不依赖应用专属 YAML、wiki links、Callout 或 URI Scheme。

标准配置应能在 Windows、macOS 和 Linux 上用任意纯文本编辑器或常见 Markdown 阅读器打开。

### Obsidian 增强（仅用户明确选择）

可以额外使用 Properties、`[[wiki links]]` 和 `> [!type]` Callout。仍建议同时保证普通 Markdown 阅读器至少能读取正文主线；关键证据和限定不能只藏在折叠 Callout 中。

Skill 不检测用户是否安装 Obsidian，不自动启动应用。用户选择增强但尚未安装时，提醒其可以安装，或立即退回标准配置；不能中断研究或只交付无法通用阅读的版本。

## 三、保存位置

1. 用户指定位置时按指定位置；
2. 未指定时优先当前工作区内一个清晰命名的子目录；
3. 当前工作区不适合时，先提出候选绝对路径并告知用户，再创建；
4. 不自行猜测桌面、文档或家目录，也不写死 `/Users/...`、`C:\...` 或 `~/...`。

## 四、Artifact-first

重度和超级重度在已确认的工作区先建立最小可恢复状态：

1. 研究状态；
2. 动态覆盖图；
3. 来源台账；
4. 主张—证据台账；
5. 解释性综合记录；
6. 深度契约；
7. 文档深读任务的原文覆盖台账；
8. 两份独立审查记录；
9. 成品骨架。

持续保存稳定成果，不把全部研究留到最后一次性生成。未经综合的摘录不得直接写入最终正文。

## 五、内部产物与最终产物

内部可以维护查询记录、候选材料、来源/主张台账、冲突缺口、综合审计和编辑备注。默认不交付这些过程文件。

最终报告必须呈现影响读者理解的公开理由：证据桥梁、关键前提、竞争解释、边界和未知。用户要求审计包时，可以整理内部台账和综合记录作为附录，但不输出私有逐步思维草稿或代理日志。

## 六、多文档架构契约

为每个计划文件内部定义：

```markdown
- 文件名：
- 核心问题：
- 在全局模型中的作用：
- 读者前置知识：
- 必须建立的概念与关系：
- 决定性证据、案例或争议：
- 读完后应能解释或判断什么：
- 明确不包含什么：
- 依赖与被依赖文档：
```

两个文件的问题和认知收益相同则合并；一个文件包含多个独立维护的问题则拆分。

### 主文档

主文档是地图和入口，不是所有子文档的重复摘要。它应定义范围、给出全局模型、说明资料日期与主要限制、链接每个模块并解释阅读理由。

推荐以 `00-` 命名只是可移植惯例，不是档位硬门；用户已有命名规范时遵循用户规范。

### 子文档

每个子文档是完整解释单元：核心问题、机制、证据桥梁、替代解释、案例/边界、含义，以及回到全局模型的出口。不能只是主文档 bullet 的扩写或代理回答的原样粘贴。

### 导航

- 使用标准 Markdown 相对链接；
- 主文档链接所有公开模块并说明用途；
- 子文档开头交代位置和必要前置；
- 交叉链接说明依赖、对照、例外或共同机制，不裸写“相关阅读”；
- 交付前检查断链、孤立文件、重复入口和循环导航。

## 七、单文档架构

单文档必须保持一条可追踪主线：中心问题、必要概念、证据与机制、竞争解释、边界、综合判断和含义。附录可以保存方法或来源索引，但不能把关键论证全部外包给附录。

若标题或段落可以任意交换仍不影响理解，说明它只是知识卡片堆叠，需要重写结构。

## 八、跨平台脚本约定

- 仅依赖 Python 3.9+ 标准库；没有可用解释器时中度以上验证状态为阻塞，不以作者自检代替；
- 路径使用 `pathlib`，文档链接统一 `/` 分隔符；
- 所有文本显式使用 UTF-8，CSV 使用 `newline=""`；
- 子进程测试使用 `sys.executable`，不写死 `python3`；
- 不检测 GUI 应用、不打开文件、不使用 shell 专属命令；
- 对中文、空格、括号、百分号、深层目录和 CRLF 建立测试；
- 错误转换为稳定诊断与退出码，不向用户抛出无解释 traceback。

## 九、交付

最终只交付通过质量门的成果，并说明：

- 主要结果与阅读入口；
- 文件绝对路径；
- Markdown 配置（standard / obsidian）；
- 资料冻结日期和对使用结果有实质影响的限制；
- 哪些自动检查已运行，哪些语义判断依赖人工审查。
